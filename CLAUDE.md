# mem0-local 프로젝트 지침

## 메모리 저장 규칙

아래 상황이 발생하면 **즉시** `add_memory` 툴을 호출해서 mem0에 저장한다.
저장 시 `user_id`는 항상 `"jay"`로 설정한다.

### 저장해야 할 내용

**인시던트 / 오류**
- 오류 발생 시: 서비스명, 증상, 원인, 해결책, 발생 시각
- 롤백 발생 시: 버전, 원인, 조치 내용
- 재발 방지를 위한 주의사항

**배포 / 변경**
- 설정값 변경 (임계값, 타임아웃, 리소스 등)
- 아키텍처 결정 및 이유
- 환경별 차이점 (개발/스테이징/프로덕션)

**패턴 / 특이사항**
- 특정 시간대 트래픽 패턴
- 반복적으로 발생하는 문제
- 서비스별 알려진 이슈

**사용자 정보**
- 작업 방식, 선호하는 도구, 코드 스타일
- 프로젝트 컨텍스트 및 담당 서비스

### 저장 형식

mem0는 LLM을 통해 텍스트에서 **사용자(Jay)와 직접 연관된 사실**만 추출한다.
단순 이벤트 로그가 아닌, **Jay의 경험·지식·책임·판단**이 담긴 문장으로 작성해야 한다.

**핵심 원칙: "Jay가 배운 것 / 경험한 것 / 알고 있는 것"으로 표현하라**

**올바른 형식 ✅**
```
Jay는 결제서비스 DB 커넥션 풀이 오전 9-10시 피크타임에 고갈되는 패턴을 경험했다.
해결책으로 max_connections를 100→200으로 늘렸고, 이후 재발하지 않았다. (2026-05-14)

Jay가 담당하는 알림서비스는 Kafka consumer lag이 10만 이상이면 장애로 이어진다는 것을 알고 있다.

Jay는 이 인프라에서 Docker 컨테이너가 Apple Silicon Metal GPU에 접근할 수 없다는 것을 알고 있다.
따라서 Ollama는 반드시 네이티브로 실행해야 한다.
```

**잘못된 형식 ❌ (필터링됨)**
```
결제서비스 DB 커넥션 풀 고갈로 타임아웃 발생. max_connections 200으로 변경.
→ 이벤트 로그처럼 보여서 mem0가 사용자 관련 사실로 인식하지 못함

Kafka consumer lag threshold: 100,000
→ 설정값만 있고 Jay와의 관계가 없어서 필터링됨
```

**체크리스트**
- [ ] "Jay는/Jay가/Jay의" 등 명시적 주어 포함
- [ ] 단순 사실 나열이 아닌 경험·판단·교훈 포함
- [ ] 날짜 또는 서비스명 컨텍스트 포함

## 메모리 검색 규칙

아래 상황에서는 먼저 `search_memory`로 관련 기록을 검색한 후 작업한다.

- 오류 분석 시작 전
- 배포 또는 설정 변경 전
- "전에 이런 문제 있었나?" 류의 질문
- 처음 보는 서비스나 컴포넌트 작업 시

---

## 프로젝트 정보

- **인프라:** Qdrant (벡터DB), Ollama (네이티브, Metal GPU), mem0 FastAPI (Docker)
- **기본 모델:** llama3-mini (빠름), 한국어 작업 시 qwen 사용
- **MCP 툴:** add_memory, search_memory, get_all_memories
